# 貢獻指南 / Contributing

這個專案收集 Agentic Coding 的**實證痛點與對策**。規則來自踩坑,不來自想像 —— 貢獻時請帶著證據來。

## 回報痛點(Issue)

最有價值的貢獻。開 Issue 時請附:

```markdown
## 環境
- Claude Code 版本 / 主模型:
- 計費模式:訂閱配額 / API per-token
- 工作型態:互動開發 / 無人值守 loop / 其他

## 痛點描述
(發生了什麼;AI 做了什麼你不想要的事)

## 頻率與代價
(偶發/每次;燒了多少 token 或時間才發現)

## 已嘗試的對策
(prompt 措辭 / 規則 / hook;有效或無效)
```

常見痛點分類(供參考,不限於此):

- **假完成**:測試綠但東西不能用;AI 沿內部指標自證完成
- **Compaction 遺失**:長對話壓縮後待辦/決策/驗收條件蒸發
- **Subagent 失控**:預設 fan-out 燒配額;低階模型靜默彙整錯誤
- **假設當事實**:使用者說「我覺得」被直接執行;外部文章主張未驗證入規
- **配置膨脹**:skill/plugin 積累,系統提示肥大,規則互相打架

## 提交 PR

### 新增或修改規則模板

每條規則必須包含:

1. **成立前提**:什麼環境/工作型態下成立?什麼情況不適用?
2. **實證來源**:踩了什麼坑、怎麼發現的(轉述即可,不需要 log 原文)
3. **可驗收**:這條規則被遵循/違反,如何從行為觀察出來?

**不收**:純理論推演的規則、從文章搬運且未在真實環境驗證的主張、綁定特定個人工具且無「如有則用」降級的寫法。

（是的,這些準則就是 skill 裡「分級對抗閘門」的精神 —— 對本 repo 的貢獻也走同一道門。）

### 其他歡迎的 PR

- 既有模板在你環境的**失效案例**(附環境差異分析)—— 和成功案例一樣有價值
- 不同計費模式/平台的實測數據
- README/SKILL 的翻譯(English translation welcome)

## 行為準則

- 證據 > 直覺,可復現 > 軼事
- 尊重環境差異:在你環境失效 ≠ 規則錯誤,寫清楚前提差異即可
- Issue/PR 使用繁中或英文皆可

## 快速開始開發

```bash
git clone https://github.com/keoy7am/YapHarnessBootstrap.git
# 技能本體只有一個檔案:
# skills/harness-bootstrap/SKILL.md
# 改完後在自己的 Claude Code 實測至少一輪 Phase 0-1,再開 PR
```
